# 第二周修改内容详解

## 1. 本次修改的目标

本次修改对应《AI客服中台MVP第二周开发详情》的全部任务，目标是**打通从用户消息输入到模型回复输出的完整问答链路**，让系统从"能存消息"升级为"能对话"。

具体交付能力：

- 用户发送消息后，系统调用大模型生成回复
- 支持 HTTP 同步问答（非流式）
- 支持 SSE 流式输出
- 支持多轮对话上下文拼接
- assistant 回复消息自动落库
- 关键链路事件写入 `message_events` 表
- 会话历史消息可查询
- LLM 调用层与业务逻辑解耦，可通过环境变量切换供应商
- 基础安全防护：频率限流、并发锁、幂等控制、内容长度限制、会话状态校验
- LLM token 用量按租户按天追踪

## 3. 核心架构设计

### 3.1 整体链路

```text
用户请求 POST /api/v1/chat
  │
  ├─ Schema 校验（content max_length=4000）
  │
  ├─ ChatGuard 前置防护
  │   ├─ check_content_length()     消息长度检查
  │   ├─ check_idempotency()        幂等检查（Redis SETNX）
  │   ├─ check_session_active()     会话状态检查
  │   ├─ check_rate_limit()         频率限流（Redis INCR）
  │   └─ acquire_session_lock()     并发锁（Redis SETNX）
  │
  ├─ ConversationOrchestrator
  │   ├─ 保存 user 消息到 messages 表
  │   ├─ ContextManager.build_context()  加载历史消息
  │   ├─ PromptBuilder.build()           拼装 system prompt
  │   ├─ EventRepo: llm_started 事件
  │   ├─ LLMProvider.chat() / chat_stream()  调用模型
  │   ├─ 保存 assistant 消息到 messages 表
  │   ├─ EventRepo: llm_finished 事件
  │   └─ TokenTracker.record_usage()     记录 token 用量
  │
  ├─ release_session_lock()
  │
  └─ 返回响应（JSON 或 SSE 流）
```

### 3.2 LLM Gateway 分层

```text
业务层（Orchestrator）
  │
  ▼
LLMProvider 抽象接口
  │  chat(ChatRequest) → ChatResponse
  │  chat_stream(ChatRequest) → AsyncIterator[ChatDelta]
  │
  ▼
OpenAIProvider 实现
  │  使用 httpx 调用 OpenAI 兼容 API
  │  捕获 httpx 异常 → 转换为 LLMError / LLMTimeoutError
  │
  ▼
任意 OpenAI 兼容供应商
  （OpenAI / DeepSeek / 通义 / Moonshot / Ollama）
```

### 3.3 安全防护层

```text
┌─────────────────────────────────────────────┐
│               ChatGuard                     │
├──────────────┬──────────┬───────────────────┤
│ 频率限流      │ 并发锁    │ 幂等控制          │
│ Redis INCR   │ Redis NX  │ Redis NX          │
│ 20次/分/用户  │ per-session│ 5分钟 TTL        │
├──────────────┼──────────┼───────────────────┤
│ 内容长度      │ 会话状态  │ Redis 降级         │
│ max 4000 字符 │ active   │ 不可用时 fail-open │
└──────────────┴──────────┴───────────────────┘
```

## 4. 新增 API 接口

### 4.1 `POST /api/v1/chat` — 问答（同步 + 流式）

**请求体：**

```json
{
  "tenant_code": "demo_tenant",
  "session_code": "s_202603260001",
  "message": {
    "type": "text",
    "content": "你好，我的订单什么时候到？"
  },
  "stream": false,
  "model": null,
  "request_id": "req_001"
}
```

**同步响应（stream=false）：**

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "message_code": "m_20260326...",
    "role": "assistant",
    "content": "您好！请提供您的订单号，我来帮您查询。",
    "finish_reason": "stop"
  },
  "trace_id": "trace_..."
}
```

**流式响应（stream=true），Content-Type: text/event-stream：**

```text
event: start
data: {"message_code":"m_20260326..."}

event: delta
data: {"content":"您好！"}

event: delta
data: {"content":"请提供您的订单号。"}

event: done
data: {"finish_reason":"stop"}
```

### 4.2 `GET /api/v1/sessions/{session_code}/messages` — 会话历史

**查询参数：** `limit`（默认 50）、`offset`（默认 0）

**响应：**

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "session_code": "s_202603260001",
    "items": [
      {
        "message_code": "m_1",
        "role": "user",
        "content": "你好",
        "status": "accepted",
        "created_at": "2026-03-26T10:00:00Z"
      },
      {
        "message_code": "m_2",
        "role": "assistant",
        "content": "您好！有什么可以帮您的吗？",
        "status": "completed",
        "created_at": "2026-03-26T10:00:02Z"
      }
    ],
    "total": 2
  },
  "trace_id": "trace_..."
}
```

## 7. Redis 使用说明

本次修改引入了 Redis 异步连接，用于以下场景：

| 用途 | Key 格式 | TTL |
|------|----------|-----|
| 频率限流 | `rate_limit:chat:{tenant}:{user_id}` | 60 秒 |
| 会话并发锁 | `chat_lock:{session_code}` | LLM 超时 + 10 秒 |
| 请求幂等 | `idempotent:chat:{request_id}` | 300 秒 |
| Token 日用量（总量） | `token_usage:{tenant}:{日期}:total` | 7 天 |
| Token 日用量（prompt） | `token_usage:{tenant}:{日期}:prompt` | 7 天 |
| Token 日用量（completion） | `token_usage:{tenant}:{日期}:completion` | 7 天 |
| Token 日调用次数 | `token_usage:{tenant}:{日期}:calls` | 7 天 |

**降级策略：** Redis 不可用时，所有依赖 Redis 的检查（限流、并发锁、幂等、用量追踪）自动跳过，不阻断正常业务。启动时会输出警告日志：

```
WARNING - Redis unavailable, guards will be disabled: ...
```

## 8. 各模块详细说明

### 8.1 LLM Gateway（`app/services/llm/`）

**抽象基类 `LLMProvider`** 定义了两个核心方法：

- `chat(ChatRequest) → ChatResponse`：非流式调用
- `chat_stream(ChatRequest) → AsyncIterator[ChatDelta]`：流式调用

**`OpenAIProvider`** 基于 `httpx` 实现 OpenAI 兼容协议：

- 使用 `httpx.AsyncClient` 发送 HTTP 请求
- 非流式：解析 `choices[0].message.content` 和 `usage`
- 流式：解析 SSE 事件流中的 `data:` 行，逐 delta 返回
- 超时 → `LLMTimeoutError`（504）
- HTTP 4xx/5xx → `LLMError`（502）
- 连接失败 → `LLMError`（502）

**工厂方法 `get_llm_provider()`** 以单例模式创建 Provider，从 Settings 读取配置。

### 8.2 上下文管理（`app/services/conversation/context_manager.py`）

- 从 `messages` 表加载指定 session 的历史消息
- 只包含 `user` 和 `assistant` 角色
- 只包含 `accepted` 和 `completed` 状态
- 按 `created_at` 升序排列
- 超过 `max_rounds * 2` 条时，只保留最近的

### 8.3 Prompt Builder（`app/services/conversation/prompt_builder.py`）

- 在上下文消息列表最前面插入 system prompt
- 默认 system prompt 包含：中文回答、不编造、简洁有条理、敏感问题转人工
- 后续可扩展注入知识片段、工具定义等

### 8.4 Conversation Orchestrator（`app/services/conversation/orchestrator.py`）

这是本次修改的**核心模块**，负责整个问答链路的编排：

1. 前置防护（Guard 检查）
2. 校验 tenant / session
3. 保存 user 消息
4. 获取并发锁
5. 加载上下文 → 拼装 prompt
6. 记录 `llm_started` 事件
7. 调用 LLM（同步或流式）
8. 保存 assistant 消息
9. 记录 `llm_finished` 事件
10. 记录 token 用量
11. 释放并发锁

流式模式下，步骤 7-10 在流完成后执行，即使流中断也会在 `finally` 块中释放锁。

### 8.5 安全防护层（`app/core/guards.py`）

`ChatGuard` 类包含 5 个检查方法，执行顺序：

1. `check_content_length()` — 同步，无 IO，最先执行
2. `check_idempotency()` — Redis SETNX
3. `check_session_active()` — 内存判断
4. `check_rate_limit()` — Redis INCR + EXPIRE
5. `acquire_session_lock()` / `release_session_lock()` — Redis SETNX + DELETE

### 8.6 Token 追踪（`app/services/token_tracker.py`）

- `record_usage()`：每次 LLM 调用后记录 prompt/completion/total tokens
- `get_daily_usage()`：查询某天的累计用量
- 数据按 `tenant_code + 日期` 维度存储，7 天自动过期

### 8.7 Event Repository（`app/db/repositories/event_repo.py`）

- `create_event()`：写入 `message_events` 表
- `event_seq` 自增逻辑：先查该 session 最大序号，然后内存缓存递增
- 支持的 event_type：`llm_started` / `llm_finished`（后续扩展 `retrieval_*` / `tool_call_*`）

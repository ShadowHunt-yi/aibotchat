# 本次修改内容详解

## 1. 本次修改的目标

这次修改的目标，不是直接做出 AI 问答能力，而是先把《AI客服中台MVP第一周开发详情》里要求的第一周工程底座搭起来，让项目具备以下能力：

- 项目可以正常启动
- 基础工程结构清晰可维护
- 数据库模型已经落地
- Alembic 迁移可执行
- 健康检查接口可用
- 会话创建接口可用
- 消息入库接口可用
- 基础测试可运行

也就是说，这次做的是“后端骨架 + 最小业务闭环”。

---

## 2. 整体架构调整

这次把仓库从“空目录骨架”推进成了真正可运行的分层项目，结构上采用：

- API 层
- Service 层
- Repository 层
- Model 层

这样拆分的目的很明确：

- API 层只负责收请求和返回响应
- Service 层只负责业务逻辑
- Repository 层只负责数据库读写
- Model 层只负责 ORM 建模

这样后面继续接 LLM、SSE、知识库、工具调用时，不会把 HTTP、业务逻辑和数据库操作混在一起。

## 3. 应用入口与生命周期

主入口文件是 `app/main.py`。

这里完成了几件关键事情：

- 创建 `FastAPI` 应用
- 注册全局路由
- 注册中间件
- 注册异常处理器
- 在应用生命周期启动时初始化日志
- 根据配置决定是否自动建表

其中 `lifespan` 里调用了：

- `configure_logging(settings)`
- `init_db()`

这意味着本地开发时，项目启动后就能直接具备可运行的数据库基础能力。

## 4. 配置系统

配置文件核心在 `app/core/config.py`。

这里使用了 `pydantic-settings` 来统一加载环境变量，并封装成 `Settings` 类。

### 4.1 这次配置里包含的内容

- 应用名称、环境、端口
- 是否自动建表
- PostgreSQL 连接参数
- Redis 连接参数
- 日志等级
- `DATABASE_URL`

### 4.2 数据库连接优先级

这里做了一个很实用的降级策略：

1. 如果配置了 `DATABASE_URL`，直接使用
2. 如果没有，就使用 `POSTGRES_*` 自动拼 PostgreSQL 连接串
3. 如果仍然不可用，就退回本地 SQLite

这样做的好处是：

- 正式环境可以直接走 PostgreSQL
- 本地开发和测试不会被数据库环境阻塞

## 5. 日志与 Trace ID

这次补了完整的日志基础设施，核心文件有：

- `app/core/logger.py`
- `app/core/middleware.py`

### 5.1 日志能力

日志里统一加入了：

- 时间
- 日志级别
- 模块名
- `trace_id`
- 具体消息

### 5.2 Trace ID 机制

为了满足文档里“链路追踪”的要求，这次实现了：

- 如果请求头里有 `X-Trace-Id`，优先使用它
- 如果没有，就自动生成一个新的 `trace_id`
- 把它保存到上下文变量里
- 记录到日志里
- 并在响应头中返回

这样后续无论是查日志、查异常，还是后面接入调用链埋点，都会更容易排查。

## 6. 请求日志中间件

`app/core/middleware.py` 里还加入了请求日志中间件。

它会记录：

- 请求方法

这部分属于第一周里“基础中间件”的内容，虽然简单，但非常重要，因为后续联调时第一时间就能看到每次请求的行为。

## 7. 统一异常处理

异常处理放在 `app/core/exception_handlers.py`。

这次实现了统一的 JSON 错误返回风格，避免每个接口各自返回不同结构。

### 7.1 自定义异常

定义了：

- `AppError`
- `NotFoundError`

`AppError` 用来承载：

- 错误消息
- 业务错误码
- HTTP 状态码

### 7.2 已处理的异常类型

目前统一处理了：

- 请求参数校验错误
- SQLAlchemy 数据库错误
- 普通 HTTP 异常
- 未知系统异常
- 自定义业务异常

### 7.3 返回结构

失败响应统一格式为：

```json
{
  "code": 40001,
  "message": "session not found",
  "data": null,
  "trace_id": "trace_xxx"
}
```

这和文档里的建议风格保持一致。

## 8. 数据库基础设施

数据库相关的基础设施主要在：

- `app/db/base.py`
- `app/db/session.py`

### 8.1 Base 与命名规范

`app/db/base.py` 中定义了：

- 全局 `Base`
- 统一约束命名规则
- `TimestampMixin`

`TimestampMixin` 给表自动提供：

- `created_at`
- `updated_at`

这样后续建表时就不需要每张表重复写一遍。

### 8.2 Session 工厂

`app/db/session.py` 中封装了：

- `create_engine_from_url`
- `create_session_factory`
- `SessionLocal`
- `get_db`
- `init_db`

其中 `get_db` 用于 FastAPI 的依赖注入，`init_db()` 用于启动时自动建表，`create_engine_from_url()` 则额外兼容了 SQLite 的线程参数，方便测试运行。

## 9. 核心数据模型

这次根据第一周文档，把核心表落成了 SQLAlchemy 模型。

### 9.1 tenants

文件：`app/db/models/tenant.py`

作用：

- 保存租户信息
- 即使当前先做单租户，也保留这个表，方便未来扩展

### 9.2 channels

文件：`app/db/models/channel.py`

作用：

- 保存接入渠道
- 比如 `demo`、`web`

这里使用了 `(tenant_id, channel_code)` 的唯一约束，避免同一租户下重复创建同名渠道。

### 9.3 users

文件：`app/db/models/user.py`

作用：

- 保存终端用户标识
- 用 `external_user_id` 对应外部用户 ID

这里使用了 `(tenant_id, external_user_id)` 的唯一约束，避免重复用户。

### 9.4 sessions

文件：`app/db/models/session.py`

作用：

- 保存会话主信息
- 包含 `tenant/channel/user` 三者关系
- 保存会话业务编码 `session_code`
- 保存状态、意图、摘要、最后消息时间等字段

这是后续多轮对话的核心主表。

### 9.5 messages

文件：`app/db/models/message.py`

作用：

- 保存消息内容
- 保存消息角色
- 保存消息类型
- 保存结构化消息扩展字段
- 可关联回复目标消息

当前第一周只实现用户消息入库，后续 assistant/tool/system 消息也都可以继续复用这张表。

### 9.6 message_events

文件：`app/db/models/message_event.py`

作用：

- 为后续流式输出、路由事件、模型事件留存结构

第一周它还没有接入业务写入逻辑，但表结构已经预留好了。这样第二周做 SSE 或 LLM 流式输出时，就不需要再重构数据库设计。

## 13. Service 层

Service 层负责业务逻辑。

### 13.1 SessionService

文件：`app/services/session_service.py`

它完成的业务链路是：

1. 根据 `tenant_code` 查租户
2. 不存在就自动创建租户
3. 根据渠道码查渠道
4. 不存在就自动创建渠道
5. 根据 `external_user_id` 查用户
6. 不存在就自动创建用户
7. 创建 session
8. 返回 `session_code`、状态、创建时间

这对应文档里 Day 4 的“自动初始化 tenant/channel/user/session 关系”。

### 13.2 MessageService

文件：`app/services/message_service.py`

它完成的业务链路是：

1. 校验 tenant 是否存在
2. 校验 session 是否存在且属于该 tenant
3. 生成 `message_code`
4. 写入用户消息
5. 更新 session 的 `last_message_at`
6. 返回消息编号、角色、状态

这对应文档里 Day 5 的“消息入库接口”。

注意这里故意没有：

- 调模型
- 生成 assistant 回复
- 写 SSE 事件流

因为这些不在第一周边界内。

### 15.1 conftest 做了什么

为了保证测试可独立运行，这里做了几个处理：

- 手动把仓库根目录加入 `sys.path`
- 为每次测试创建独立 SQLite 数据库
- 把测试数据库文件放在仓库内 `.pytest_tmp` 目录
- 用 FastAPI `dependency_overrides` 替换掉正式环境的 `get_db`

之所以不用系统默认临时目录，是因为当前环境对系统临时目录权限比较严格，仓库内自建测试目录会更稳定。

- 会话不存在时报错
